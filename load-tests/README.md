# Нагрузочные тесты

Здесь лежат скрипты для измерения производительности webhook-сервиса. Цель — найти максимальный RPS, который сервис выдерживает без деградации, и посмотреть как он себя ведёт под экстремальной нагрузкой.

## Файлы

| Файл | Что делает |
|------|-----------|
| `run-capacity.sh` | Главный скрипт запуска. Поднимает сервис, прогоняет все тесты по матрице конфигураций, сохраняет результаты. |
| `capacity.js` | k6-сценарий для ступенчатого теста и soak. |
| `spike.js` | k6-сценарий для spike теста. |
| `ci-check.js` | Быстрая smoke-проверка для CI, 30 секунд. |
| `mock-target.js` | Простой HTTP-сервер, имитирующий upstream для proxy-вебхуков. |
| `compute-sustainable.js` | Вычисляет максимальный устойчивый RPS по результатам ступенчатого теста. Используется внутри `run-capacity.sh`. |
| `gen-capacity-report.js` | Собирает все JSON-результаты в один отчёт `CAPACITY.md`. |
| `lib/common.js` | Общие утилиты для k6-скриптов: настройки подключения, создание вебхуков, сбор статистики. |
| `CAPACITY.md` | Готовый отчёт с результатами последнего прогона. |
| `results/capacity/` | Сырые JSON-файлы каждого прогона (не коммитятся в git). |

## Требования

- `docker compose`
- `k6` версии 0.50 и выше
- `node` версии 18 и выше
- `curl`, `taskset` (пакет `util-linux`)

## Запуск

```bash
cd repo/load-tests

# Полный прогон — занимает 2–3 часа на 12-ядерной машине
./run-capacity.sh

# Быстрая проверка, около 10 минут
QUICK=1 ./run-capacity.sh
```

Результат сохраняется в `CAPACITY.md`.

## Как это работает

### Ступенчатый тест

Нагрузка подаётся ступенями: сначала 1000 запросов в секунду, потом 2000, потом 4000 и так далее. Каждая ступень длится фиксированное время. На каждой ступени фиксируется: реально достигнутый RPS, задержки p50/p95/p99, процент ошибок.

Ступень считается пройденной, если:
- реально достигнутый RPS не ниже 99% от целевого
- процент ошибок не превышает 0.1%
- не было отброшенных запросов (dropped iterations)
- p99 укладывается в бюджет задержки сценария

Наивысшая пройденная ступень — это и есть **максимальный устойчивый RPS**.

Важно: используется открытая модель нагрузки (constant arrival rate). Это значит, что k6 всё равно пытается держать заданный RPS, даже если сервер начинает тормозить. Благодаря этому насыщение видно честно: растёт p99, появляются отброшенные запросы.

### Soak-тест

После нахождения максимального RPS сервис нагружается на уровне ~80% от него в течение нескольких минут. Это проверяет стабильность: нет ли утечек памяти, не растёт ли задержка со временем.

### Spike-тест

Резкий скачок нагрузки вдвое выше максимального устойчивого RPS, затем возврат к базовому уровню. Проверяет, как сервис переживает всплески трафика: корректно ли сбрасывает лишние запросы и восстанавливается ли после спайка.

### Прогрев

Перед каждой серией ступенчатых тестов делается короткий прогрев — ненагружаемый прогон, который инициализирует JIT-компиляцию и заполняет пул соединений. Это нужно, чтобы первая измеряемая ступень не выглядела хуже из-за cold start эффекта.

## Матрица конфигураций

Тесты прогоняются для всех комбинаций:
- **Кэш:** включён / выключен
- **Verticles:** 1, 2, 4 (число event loop потоков)
- **Сценарии:** `simple` (горячий путь без прокси), `proxy` (с проксированием), `crud` (создание/чтение/удаление вебхуков)

## Настройка через переменные окружения

| Переменная | По умолчанию | Описание |
|-----------|-------------|----------|
| `CACHE_MODES` | `on off` | Режимы кэша |
| `VERTICLE_SET` | `1 2 4` | Число verticles |
| `WORKLOADS` | `simple proxy crud` | Сценарии нагрузки |
| `STEPS_SIMPLE` | `1000…24000` | Ступени RPS для `simple` |
| `STEPS_CRUD` | `500…8000` | Ступени RPS для `crud` |
| `STEP_DURATION` | `25s` | Длительность одной ступени |
| `SOAK_DURATION` | `5m` | Длительность soak-теста |
| `BUDGET_SIMPLE` | `50 мс` | Допустимый p99 для `simple` |
| `BUDGET_PROXY` | `100 мс` | Допустимый p99 для `proxy` |
| `BUDGET_CRUD` | `150 мс` | Допустимый p99 для `crud` |
| `CLEAN=1` | — | Удалить старые результаты и начать заново |
| `QUICK=1` | — | Сокращённая матрица для быстрой проверки |

## Повторный запуск и возобновление

Если тест прервался, можно запустить его снова — уже готовые JSON-файлы будут пропущены, прогон продолжится с того места, где остановился.

Чтобы начать полностью заново:

```bash
CLEAN=1 ./run-capacity.sh
# или вручную
rm -rf results/capacity/
./run-capacity.sh
```
